## Overview

This use case describes the complete onboarding workflow for new client portal users, from initial invitation through first successful login and portal orientation. The onboarding process is designed to be self-service while ensuring proper security, data collection, and user education.

**Key Objectives:**
- Secure account creation with proper authentication
- Complete user profile setup
- Portal orientation and feature discovery
- Organization structure configuration (for admins)
- Integration setup (SSO, calendar, etc.)

---

## Actors

**Primary:**
- Client Portal User (new user accepting invitation)
- Client Admin (user with admin privileges)

**Secondary:**
- InTime Account Manager (sends initial invitation)
- InTime System Admin (assists with SSO setup)

**System:**
- Email delivery system
- SSO provider (if applicable)
- Calendar integration service
- Audit logging service

---

## Preconditions

1. Client account exists in InTime system
2. Account Manager has configured basic client settings
3. User has valid business email address
4. Email delivery is functional
5. Client has active contract with InTime

---

## Trigger

**Primary Trigger:**
InTime Account Manager sends portal invitation to client user

**Alternative Triggers:**
- Client Admin invites additional team members
- SSO auto-provisioning creates new user account
- Bulk user import from HR system

## Alternative Flows

### ALT-1: SSO-Based Onboarding

**Flow:**
1. User clicks SSO login from client portal
2. Redirects to identity provider (Azure AD, Okta, etc.)
3. User authenticates with corporate credentials
4. Identity provider redirects back with SAML assertion
5. InTime validates assertion and auto-creates/updates user
6. User skips password creation, goes directly to profile setup
7. MFA may be handled by identity provider

### ALT-2: Bulk User Import

**Flow:**
1. Client Admin uploads CSV with user list
2. System validates email addresses and role assignments
3. System sends invitation emails in batch
4. Each user follows standard onboarding flow
5. Admin dashboard shows import progress and status

**CSV Format:**
```csv
email,firstName,lastName,role,department
john.smith@acme.com,John,Smith,hiring_manager,Engineering
jane.doe@acme.com,Jane,Doe,client_admin,HR
bob.jones@acme.com,Bob,Jones,procurement_manager,Procurement
```

### ALT-3: Mobile-First Onboarding

**Flow:**
1. User opens invitation email on mobile device
2. Redirected to mobile-optimized onboarding flow
3. Simplified single-page registration
4. Photo upload via camera
5. Optional: "Continue on desktop" for full setup
6. Basic access granted immediately, full features on desktop

## Business Rules

### BR-1: Invitation Expiry
- Standard invitations expire after 7 days
- Client Admin can customize expiry (1-30 days)
- Expired invitations must be resent
- Maximum 3 invitation attempts before manual review

### BR-2: Password Requirements
- Minimum 12 characters
- Must include: uppercase, lowercase, number, special character
- Cannot match common password lists (10,000+ passwords)
- Cannot include user's name or email
- Cannot reuse last 5 passwords

### BR-3: MFA Requirements
- Mandatory for all Client Admin users
- Mandatory for users with timesheet approval rights
- Optional but recommended for standard users
- Cannot be disabled once enabled (security policy)
- Recovery codes must be saved during setup

### BR-4: Email Verification
- Email must be verified before account activation
- Verification link expires after 24 hours
- Maximum 3 verification email requests per hour
- Must use corporate email domain (no free email providers for enterprise)

### BR-5: Profile Completeness
- Minimum required: name, email, phone, timezone
- Job title required for hiring managers
- Department required for approval workflow routing
- Profile photo optional but recommended

### BR-6: Role Assignment
- Initial role assigned by inviter
- Role determines available features and permissions
- Only Client Admin can change user roles
- Role changes require re-authentication

### BR-7: Data Privacy
- User must accept Terms of Service
- User must accept Privacy Policy
- Acceptance is tracked with timestamp and IP
- New ToS versions require re-acceptance

### BR-8: SSO Configuration
- Requires Enterprise plan
- Must be configured by Client Admin
- Requires IT department coordination
- Test connection required before enforcement
- Fallback password login available for 30 days

## Security

### Data Protection

**In Transit:**
- TLS 1.3 encryption
- HSTS enforcement
- Certificate pinning (mobile)

**At Rest:**
- AES-256 encryption for passwords
- Bcrypt hashing (12 rounds)
- Encrypted MFA secrets
- Encrypted backup codes

**Session Security:**
- HTTPOnly cookies
- Secure flag set
- SameSite=Strict
- CSRF tokens
- Session timeout: 30 minutes idle

### Privacy

**Data Minimization:**
- Only collect required data
- Optional fields clearly marked
- Purpose disclosed for each field

**Consent:**
- Explicit ToS acceptance
- Privacy policy agreement
- Marketing opt-in separate

**Right to Erasure:**
- Account deletion available
- Data export before deletion
- 30-day grace period

### Audit Trail

**Logged Events:**
- Invitation sent (who, when, to whom)
- Account created (timestamp, IP)
- Password set (timestamp, complexity met)
- MFA enabled (method, timestamp)
- Profile updated (fields changed)
- Integration connected (provider, timestamp)
- First login (timestamp, IP, user agent)

**Retention:**
- 90 days in hot storage
- 7 years in cold storage
- Compliance requirement
