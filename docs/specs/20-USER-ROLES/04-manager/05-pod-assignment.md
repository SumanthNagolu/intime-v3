# UC-MGR-005: Pod Assignment and Formation

**Version:** 1.0
**Last Updated:** 2025-11-30
**Role:** Pod Manager / Regional Director
**Status:** Canonical Reference

---

## 1. Overview

Pods are flexible, project/client/domain/region-based teams in InTime OS. Unlike traditional fixed departments, pods are formed dynamically based on business needs and can be created, merged, split, or disbanded as requirements change. This document defines how pods are assigned, formed, and managed.

**Key Principle:** Pods = Teams. Pods are the fundamental organizational unit for ICs.

---

## 2. Actors

- **Primary:** Regional Director (creates/disbands pods)
- **Secondary:** Pod Manager (manages pod membership and IC assignments)
- **Supporting:** COO (approves major pod restructuring), HR (handles employment changes)
- **System:** InTime Platform (enforces pod assignments, RACI rules)

---

## 3. Pod Formation Triggers

### 3.1 When to Create a New Pod

| Trigger | Threshold | Example |
|---------|-----------|---------|
| **Geographic Expansion** | New city/region opened | Opening NYC office - create NYC Recruiting Pod |
| **Client Dedication** | Strategic account requiring dedicated team | Fortune 500 client signs $5M+ annual contract |
| **Vertical Specialization** | Critical mass of work in domain | 10+ healthcare IT jobs active - create Healthcare Pod |
| **Pod Capacity Exceeded** | Pod at max capacity | Recruiting Pod has 12 ICs (over 10 max) |
| **Skills Gap** | Specialized skillset needed | Cybersecurity roles require dedicated pod |
| **Service Line Launch** | New business offering | Launching Executive Search - create Executive Pod |
| **Performance Isolation** | Underperforming pod segment | Split high/low performers to focus coaching |
| **Merger/Acquisition** | Acquiring team/company | Acquiring 15-person bench sales team |

**Typical Pod Lifespans:**
- **Client-Based Pod:** 1-3 years (contract duration)
- **Geographic Pod:** 3-10+ years (until market exits)
- **Vertical Pod:** 2-5 years (industry trends)
- **Project Pod:** 6-18 months (project lifecycle)

## 4. Pod Types and Structure

### 4.1 Pod Types

| Pod Type | Formation Basis | Typical Size | Manager Type | Example |
|----------|----------------|--------------|--------------|---------|
| **Recruiting Pod** | Client/vertical/geography | 5-10 ICs | Recruiting Manager | "Enterprise SaaS Recruiting Pod" |
| **Bench Sales Pod** | Geography/visa type | 3-8 ICs | Bench Manager | "West Coast Bench Sales Pod" |
| **TA Pod** | Service offering | 2-5 ICs | TA Manager | "Corporate Training Lead Gen Pod" |
| **Strategic Account Pod** | Single major client | 3-10 ICs | Account Manager | "Google Strategic Account Pod" |
| **Vertical Pod** | Industry domain | 5-12 ICs | Vertical Manager | "Financial Services Recruiting Pod" |
| **Regional Pod** | Geography | 8-15 ICs | Regional Manager | "NYC/NJ Recruiting Pod" |

### 4.2 Pod Composition (RACI Structure)

Every pod MUST have:
- **1 Pod Manager** (Consulted on all pod objects)
- **2-12 Individual Contributors** (ICs) - Responsible/Accountable on objects
- **Secondary Stakeholders:** Regional Director (Informed), COO (Informed)

## 8. Business Rules

### 8.1 Pod Formation Rules

1. **Manager Requirement:** Every pod MUST have exactly 1 primary pod manager
2. **Minimum Size:** Pod must have at least 2 ICs (manager + 1 IC is not a pod)
3. **Maximum Size:** Pod should not exceed 12 ICs (split if larger)
4. **Regional Alignment:** Pod must belong to a region with a Regional Director
5. **Budget Approval:** Pods requiring new headcount need Regional Director + CFO approval
6. **Charter Required:** Every pod must have a documented charter/mission
7. **Scope Definition:** Pod scope must be clearly defined (cannot be "everything")

### 8.2 Pod Manager Assignment Rules

1. **Single Primary:** Each pod has exactly 1 primary manager
2. **Capacity Limit:** Manager should not manage more than 2 pods (ideally 1)
3. **Expertise Match:** Manager should have domain expertise for pod type
4. **RACI Auto-Assignment:** Manager is automatically assigned as Consulted (C) on all pod objects
5. **Permissions:** Manager automatically gets pod-scoped permissions upon assignment

### 8.3 IC Assignment Rules

1. **Single Pod:** IC can belong to only 1 pod at a time
2. **Notification:** IC must be notified of pod assignment
3. **Object Transfer:** When IC changes pods, their objects can transfer or stay (defined per object)
4. **Training:** IC must complete pod-specific onboarding
5. **RACI Preservation:** IC's RACI assignments on objects are preserved across pod changes

## 9. Integration Points

### 9.1 tRPC Procedures

```typescript
// Pod creation procedure
pods.create({
  input: {
    name: string;
    type: PodType;
    region: string;
    managerId: string;
    charter: string;
    scopeClients?: string[];
    scopeRegions?: string[];
    scopeVerticals?: string[];
    targetPlacementsPerIC: number;
    targetRevenueQuarterly?: number;
    targetRevenueAnnual?: number;
    targetMarginPct?: number;
  };
  output: {
    podId: string;
    success: boolean;
  };
});

// IC assignment to pod
pods.assignMember({
  input: {
    podId: string;
    userId: string;
    roleInPod: 'manager' | 'ic' | 'secondary';
  };
  output: {
    success: boolean;
  };
});
```

### 9.2 System Side Effects

**On Pod Creation:**
1. Create pod record in database
2. Assign pod manager with Consulted (C) role
3. Create pod dashboard and metrics
4. Set up pod-specific permissions (RLS policies)
5. Notify Regional Director and COO
6. Create audit log entry

**On IC Assignment to Pod:**
1. Update user.pod_id
2. Update pod member count
3. Auto-assign RACI for new pod objects
4. Transfer or reassign existing objects (based on policy)
5. Notify IC and pod manager
6. Update pod dashboard
7. Create audit log entry
